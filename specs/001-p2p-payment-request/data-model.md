# Data Model: Lovie P2P Payment Request

## Overview

The feature is built around three persisted entities in PostgreSQL plus one volatile session model
used only by the Fastify process. The API exposes a derived read model that enriches payment
requests with participant summaries and expiry metadata.

## Entity: User

| Field | Type | Rules |
|-------|------|-------|
| `user_id` | `uuid` | Primary key generated by `gen_random_uuid()` |
| `email` | `varchar(255)` | Required, unique, used for login and recipient lookup |
| `phone` | `varchar(20)` | Required, unique, used for recipient lookup |
| `status` | `varchar(20)` | Defaults to `Active` in the TypeScript domain model |
| `created_at` | `timestamptz` | Creation timestamp |

**Relationships**

- One `User` owns exactly one `Account`.
- One `User` can appear as `requester_id` on many `PaymentRequest` rows.
- One `User` can appear as `recipient_id` on many `PaymentRequest` rows.

## Entity: Account

| Field | Type | Rules |
|-------|------|-------|
| `account_id` | `uuid` | Primary key generated by `gen_random_uuid()` |
| `user_id` | `uuid` | Required, unique foreign key to `users.user_id` |
| `balance` | `numeric(19,2)` | Required, defaults to `0.00`, must remain `>= 0` |
| `currency` | `varchar(3)` | Required, defaults to `USD` |
| `version` | `integer` | Required, defaults to `0`, used for optimistic updates |
| `updated_at` | `timestamptz` | Timestamp updated after balance changes |

**Invariants**

- Each user can have only one account in the assignment build.
- Balance must never become negative.
- Successful payment operations increment the account version used during update checks.

## Entity: PaymentRequest

| Field | Type | Rules |
|-------|------|-------|
| `request_id` | `uuid` | Primary key generated by `gen_random_uuid()` |
| `requester_id` | `uuid` | Required foreign key to `users.user_id` |
| `recipient_id` | `uuid` | Required foreign key to `users.user_id` |
| `amount` | `numeric(19,2)` | Required, `> 0`, max `999,999.99` |
| `currency` | `varchar(3)` | Required, assignment-constrained to `USD` |
| `status` | `varchar(20)` | Required, one of `PENDING`, `PAID`, `DECLINED`, `EXPIRED`, `CANCELLED` |
| `memo` | `varchar(280)` | Optional, trimmed, must not contain HTML-like content |
| `created_at` | `timestamptz` | Creation timestamp |
| `updated_at` | `timestamptz` | Updated whenever status changes |
| `expires_at` | `timestamptz` | Required, set to seven days after creation |

**Invariants**

- `requester_id` and `recipient_id` must be different users.
- Only `PENDING` requests may transition to another state.
- `PAID`, `DECLINED`, `EXPIRED`, and `CANCELLED` are terminal states.

## Volatile Model: Session

| Field | Type | Rules |
|-------|------|-------|
| `user_id` | `string` | Current authenticated user |
| `email` | `string` | Lower-cased login email |
| `created_at` | `number` | Process-local timestamp in milliseconds |

**Notes**

- Sessions are stored in memory in the Fastify process.
- They are valid only until the process restarts.

## Derived Read Model: PaymentRequestDetails

The API enriches persisted request data with participant summaries and derived expiry fields:

| Field | Source |
|-------|--------|
| `requester` | Joined from `users` as `{ user_id, email, phone }` |
| `recipient` | Joined from `users` as `{ user_id, email, phone }` |
| `shareable_link` | Derived from `APP_BASE_URL + /requests/{request_id}` |
| `is_expired` | Derived from status or `expires_at <= now` |
| `expires_in_seconds` | Derived countdown for UI rendering |

## State Transitions

| From | Action | To | Guard |
|------|--------|----|-------|
| `PENDING` | Pay | `PAID` | Recipient only, sufficient funds, request not expired |
| `PENDING` | Decline | `DECLINED` | Recipient only, request not expired |
| `PENDING` | Cancel | `CANCELLED` | Requester only, request not expired |
| `PENDING` | Expiry sweep | `EXPIRED` | `expires_at <= now` |

No other transition is valid.

## Validation Rules

- Email must match the repo's email validation regex.
- Phone identifiers are normalized to E.164-like values where possible.
- Amount must parse to a positive number with up to two decimal places.
- Currency is normalized to uppercase and only `USD` is accepted.
- Memo is trimmed, optional, limited to 280 characters, and rejects HTML-like content.

## Indexing Notes

- `users` is indexed by lowercase email and phone.
- `payment_requests` is indexed by requester, recipient, status, and expiry time.
- These indexes support the list, search, and expiry-heavy reads defined in the feature.
